# Feature: M2.3 — Frontend Catalog Tab (Public Browse + Skill Detail)

The following plan should be complete, but it is important that you validate documentation, codebase patterns, and task sanity before you start implementing. M0–M3 backend endpoints and the M2.2 admin curator UI are merged. This plan adds the **public discovery surface** in `frontend/` so contributors, managers, and agent operators can browse the published catalog without `curl`-ing `GET /v1/skills`.

Pay special attention to:

- **Mirror the existing frontend conventions exactly.** Today the codebase uses **Next.js 14 App Router + Client Components with `useEffect`/`useState`** (see `frontend/app/my-submissions/page.tsx:1-127` and `frontend/app/admin/queue/page.tsx`). The M2.2 plan introduced `useResource` (`frontend/lib/hooks/useResource.ts`) as the SWR-shaped wrapper over `api`. **Use `useResource` for both the catalog list and the detail page.** Do not add `swr` to `package.json`. Do not migrate `my-submissions/page.tsx` in this plan — that's deliberately out of scope.
- **Filtering, sorting, and search are client-side only.** Per the user decision for this milestone, `GET /v1/skills` stays unchanged: it returns all approved skills, with the existing 60s Redis list cache (`key_cache_list()` in `backend/core/redis.py`). The catalog page fetches the full list once, then filters/sorts/searches in-memory. This is correct for v1 catalog sizes (dozens-to-low-hundreds of skills) and matches the AGENTS.md §3 storage split — Cosmos is the truth, Redis is the cache, no per-filter cache keys are introduced. **Do NOT add `category`/`tag`/`q`/`cursor`/`limit` query params to `GET /v1/skills` in this milestone**, even though the PRD §10 lists them as the eventual shape. That is a separate plan when catalog size warrants it.
- **The detail page needs the SKILL.md body, which the current `SkillListItem` DTO does not carry.** This plan introduces a new `SkillDetail` DTO (`SkillListItem` + `skill_md_text: str`) and returns it from `GET /v1/skills/{skill_id}`. This is a narrow additive change to one endpoint and one Pydantic model. The Cosmos `SkillDoc` already has `skill_md_text` (`backend/models/skill.py:67`); the catalog service already loads the full doc (`backend/services/catalog.py:69-100`). The change is plumbing only — no new Cosmos write paths, no new cache keys, no new auth surface.
- **Human downloads ARE a usage signal.** Per the user decision, when a human clicks "Download tar.gz" from the catalog UI the frontend MUST `POST /v1/skills/{skill_id}/usage` with `loader_id="web-ui:<email>"` so the curator's 30/90-day staleness counters reflect human exploration as well as agent loads. The POST is fire-and-forget (no `await` blocking the redirect) and tolerates errors silently — the user must always get their bundle even if usage telemetry fails. See "Patterns to Follow → Download flow" below.
- **Auth: the Catalog tab is visible to everyone in the nav.** `GET /v1/skills` already requires a principal via `get_principal` (`backend/api/skills.py:33-40`). In stub mode anyone gets `anon@org`; in OIDC mode the user is redirected to login by the typed client (`frontend/lib/api/client.ts:53` already handles this). No new auth gate is needed in this plan. Do NOT add a `RequireAuth` wrapper.
- **Storage placement and never-delete are not in scope.** This UI is read-only against the existing catalog endpoints (`GET /v1/skills`, `GET /v1/skills/{id}`, `GET /v1/skills/{id}/download`, `POST /v1/skills/{id}/usage`). Nothing in this plan writes Cosmos or Blob. The four Redis rules (AGENTS.md §4) are upheld server-side and the frontend touches nothing in Redis.
- **The Catalog is NOT the admin Skills view.** `frontend/app/admin/curator/skills/page.tsx` (from M2.2) is the admin-only fleet view with pin/unpin/restore actions. The new Catalog tab is the public discovery surface — read-only, no admin affordances, no archived skills, no pin toggles. Different audiences, different routes, do not merge.

## Feature Description

A new `/catalog` route in the Next.js frontend that gives every signed-in user a single page for browsing the published catalog of agent skills. The page is read-only, client-side-filtered, and composed of three building blocks:

1. **A skill grid** at `/catalog` showing every `status="approved"` skill as a card with name, summary, category, tags, quality score, version, uploader, and a "View" link. Cards reuse the existing `ms-card` design language and `StatusBadge`/`ms-chip` primitives.
2. **A filter bar** above the grid for filtering by category (single-select dropdown derived from the data), tags (multi-select chips), free-text search (matches name + description + summary + tags), and sort (newest first / quality score desc / name A→Z). All filtering happens client-side over the list returned by `GET /v1/skills`.
3. **A skill detail page** at `/catalog/[skillId]` showing the rendered SKILL.md body (via `MarkdownView`), full classifier metadata, version, uploader, approved-at timestamp, bundle checksum + size, and a primary "Download tar.gz" button that hits `GET /v1/skills/{skill_id}/download` (which 307-redirects to a signed SAS URL) and fires a usage event in parallel.

## User Story

As a **contributor** I want to browse what skills already exist before I submit a new one, so I avoid creating a duplicate that the classifier will flag.

As a **manager** I want to filter approved skills by category and search by keyword, so I can confirm coverage of a domain before approving yet another submission in that space.

As an **agent operator integrating Hermes Agent** I want a human-readable page per skill with the rendered SKILL.md, the bundle checksum, and a direct download link, so I can verify the skill before wiring my runtime's API-key consumer to its `skill_id`.

As an **admin reviewing the curator's 30/90-day staleness counters** I want human downloads from the UI to count as usage signals, so a skill that humans actively reference doesn't get auto-archived just because no agent runtime has POSTed `/usage` recently.

As an **unauthenticated visitor in OIDC mode** I want the Catalog tab to redirect me cleanly to login on first fetch (no broken page), so the experience matches every other protected route in the app.

## Problem Statement

Today the published catalog is queryable only via the REST API: contributors and managers have no UI surface to browse approved skills. The only frontend pages that touch the `skills` data are `my-submissions` (per-uploader view) and `admin/queue` (pending-review queue). This:

1. Forces contributors to grep through `GET /v1/skills | jq` to check for duplicates before uploading, which they don't do, so the classifier's `duplicate_candidates` flag becomes the only defense and the manager-override rate creeps up.
2. Hides the catalog from agent operators who are exactly the consumers the public API was built for. Operators have to read the OpenAPI docs to discover `GET /v1/skills/{id}/download`, which puts a "developer-only" tax on the product's main value proposition.
3. Leaves the curator's staleness signal one-sided: agent runtimes POST `/usage` on load, but humans browsing or reading SKILL.md never generate a signal, so the curator's "no loads in 30 days" trigger archives skills humans actively use.
4. Makes the public catalog API feel like an afterthought relative to the admin/queue and admin/curator surfaces that already have polished UIs.

## Solution Statement

Ship a focused `/catalog` section that:

1. Adds a single Client Component page at `frontend/app/catalog/page.tsx` rendering the approved-skill list via `useResource(api.catalog.list)`, with client-side filtering and sorting in `useMemo`. Reuses `StatusBadge` and the `ms-card` / `ms-chip` design system; no new design tokens.
2. Adds `frontend/app/catalog/[skillId]/page.tsx` for the detail view, fetching via `useResource(api.catalog.get)` and rendering the SKILL.md body through a new `<MarkdownView>` (the M2.2 plan already created this component for the curator runs page — reuse it; do not create a second copy).
3. Adds a "Download tar.gz" button on the detail page that:
   - Navigates the browser to `${BASE}/v1/skills/${skillId}/download` (anchor `href`, browser follows the 307 to the signed SAS URL, no JS download streaming).
   - In parallel, fires `void api.catalog.reportUsage(skillId, { loader_id: 'web-ui:<email>' })` (fire-and-forget, errors logged to `console.warn`, never surfaced to the user).
4. Adds one narrow backend change: a new `SkillDetail` DTO in `backend/models/api.py` (extends `SkillListItem` with `skill_md_text: str`), used as the response model for `GET /v1/skills/{skill_id}`. `catalog.get_skill` already loads the full `SkillDoc` with `skill_md_text`, so the change is response-mapping only. The Redis item cache key is unchanged; old cache entries serialize cleanly into the new model because `skill_md_text` defaults to `""` in `SkillDoc`.
5. Adds a "Catalog" link to the top nav in `frontend/app/layout.tsx` (between "Upload" and "My submissions"), visible to all signed-in users — no admin gate, no role check.
6. Ships **no new backend filter/cursor/pagination parameters** in this milestone. The client-side filter approach is sized for v1; a future milestone (e.g. M4 "catalog scale") can revisit when the corpus exceeds ~500 approved skills.

```tsx
// frontend/components/catalog/DownloadButton.tsx
"use client";
import { api } from "@/lib/api/client";

export function DownloadButton({ skillId }: { skillId: string }) {
  const onClick = () => {
    const email =
      typeof window !== "undefined"
        ? window.localStorage.getItem("x-user-email") ?? "anon@org"

      // the SAS URL also carries Content-Disposition.

      Download tar.gz
```

**Client-side filter pattern:**

```ts
// frontend/lib/catalog/filter.ts
import type { SkillListItem } from "@/lib/api/types";

export interface FilterArgs {
  category: string | null; // null = "all"
  tags: string[]; // empty = "all"
  q: string; // empty = "no search"
}

export type SortKey = "newest" | "quality" | "name";

export function filterSkills(
  skills: SkillListItem[],
  { category, tags, q }: FilterArgs,
): SkillListItem[] {
  const needle = q.trim().toLowerCase();
  return skills.filter((s) => {
    if (category && s.classification?.category !== category) return false;
    if (tags.length > 0) {
      const skillTags = new Set(s.classification?.tags ?? []);
      if (!tags.every((t) => skillTags.has(t))) return false;
    }
    if (needle) {
      const hay = [
        s.name,
        s.description,
        s.classification?.summary ?? "",
        ...(s.classification?.tags ?? []),
      ]
        .join(" ")
        .toLowerCase();
      if (!hay.includes(needle)) return false;
    }
    return true;
  });
}

export function sortSkills(
  skills: SkillListItem[],
  key: SortKey,
): SkillListItem[] {
  const copy = [...skills];
  if (key === "newest") {
    copy.sort(
      (a, b) =>
        new Date(b.uploaded_at).getTime() - new Date(a.uploaded_at).getTime(),
    );
  } else if (key === "quality") {
    copy.sort(
      (a, b) =>
        (b.classification?.quality_score ?? 0) -
        (a.classification?.quality_score ?? 0),
    );
  } else {
    copy.sort((a, b) => a.name.localeCompare(b.name));
  }
  return copy;
}

export function extractCategories(skills: SkillListItem[]): string[] {
  const set = new Set<string>();
  for (const s of skills) {
    const c = s.classification?.category;
    if (c) set.add(c);
  }
  return Array.from(set).sort();
}

export function extractTags(skills: SkillListItem[]): string[] {
  const set = new Set<string>();
  for (const s of skills) {
    for (const t of s.classification?.tags ?? []) set.add(t);
  }
  return Array.from(set).sort();
}
```

### Edge Cases

- Skill with `classification === null` (pre-classification approve flow doesn't exist today, but defensive code wins): card must not crash; filters must not throw.
- Skill with empty tags list: filter UI must not show an "All tags (0)" empty chip group.
- Skill with `skill_md_text === ""` (legacy doc, or upload that somehow had an empty body): detail page must show a "No SKILL.md body" placeholder rather than a blank prose block.
- 404 on detail page: friendly message with back-link, no thrown error to the React boundary.
- OIDC mode unauthenticated: first fetch triggers the MSAL redirect; on return, the catalog loads normally.
- Stub mode `anon@org`: the catalog list still returns approved skills; the download `loader_id` is `web-ui:anon@org`.
- Redis down: catalog list still works via Cosmos fallback (already tested by `test_redis_down_fallback.py`); no new test needed but verify the fallback path is exercised by reading the existing test.
- Bundle missing on an approved skill (impossible per current invariants, but defensive): detail page hides the Download button.
- Click Download twice rapidly: two usage events fire, both succeed independently; no idempotency required (the curator counts events, doesn't dedupe).

## NOTES

**Why client-side filtering for v1.** Per the user decision, this milestone does not extend `GET /v1/skills` with `category`/`tag`/`q`/`cursor`/`limit` query parameters (which the PRD §10 reserves). The catalog at v1 scale is dozens to low-hundreds of skills; a single 60s-cached list response is well under 100KB of JSON and filtering is instant in `useMemo`. The follow-up milestone for backend-side filtering should land when (a) the catalog crosses ~500 approved skills, or (b) a real metric shows the initial-fetch payload becomes a perf problem on mobile/slow networks. At that point the same `useResource` hook can swap from `api.catalog.list()` to a thin `api.catalog.search(params)` without touching the page composition.

**Why human downloads count as usage signals.** Per the user decision, when a human clicks Download from `/catalog/{id}`, the frontend POSTs `/v1/skills/{id}/usage` with `loader_id="web-ui:<email>"`. This means human exploration prevents the curator's 30/90-day staleness window from archiving a skill that humans actively reference, even if no agent runtime has loaded it recently. The trade-off: the curator's "loaders_30d" counter blends human + agent activity, which slightly muddies the "is this skill being used by automation?" signal. Acceptable for v1; a future plan can split the counters by `loader_id` prefix (`web-ui:` vs `hermes:` vs `api-key:`) if the curator wants to weight them differently.

**Why no URL-state for filters.** Filter state lives in component state, not the URL. Reasons: (a) no shareable-link requirement in the user stories above, (b) avoids `useSearchParams` + `router.replace` complexity, (c) avoids a re-render cascade per keystroke. A future plan can wire URL state if shareable filtered views become a need.

**Why no `RequireAuth` wrapper.** The backend already enforces auth via `get_principal` on every catalog endpoint. The typed client (`frontend/lib/api/client.ts:53`) already handles the OIDC redirect on 401. In stub mode the request succeeds as `anon@org`. Adding a frontend gate would either duplicate the backend check (drift risk) or block the stub path (DX regression). Not worth it.

**Why no Vitest setup as a hard prerequisite.** The M2.2 plan introduces Vitest. If M2.2 is fully shipped, frontend unit tests for `filter.ts` are mandatory. If M2.2 is partially shipped or not shipped, the implementer should document the gap and ship the integration test (Task 3) only. Vitest setup is **explicitly not** added by this plan — that's M2.2's job.

**Confidence in one-pass success.** 8/10. Risks: (1) the dynamic-route `params` shape in Next 16 — verify against the existing `runs/[runId]/page.tsx` before writing the detail page. (2) Whether `MarkdownView` already exists from M2.2 — if not, the implementer creates a 12-line wrapper. (3) Whether Vitest is configured in `frontend/` — gracefully degraded by deferring the unit test (Task 7).
