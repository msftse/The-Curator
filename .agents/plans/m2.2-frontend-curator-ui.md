# Feature: M2.2 — Frontend Admin UI for the Curator

The following plan should be complete, but it is important that you validate documentation, codebase patterns, and task sanity before you start implementing. M0/M1/M2 (deterministic curator) and M3 (LLM review proposals) backend endpoints are merged. This plan adds the **operator surface** in `frontend/` so an admin can drive the curator without `curl`.

Pay special attention to:

- **Mirror the existing frontend conventions exactly.** The codebase today uses **Next.js 14 App Router + Client Components with `useEffect`/`useState`** (see `frontend/app/admin/queue/page.tsx:1-119`). SWR is **not** a dependency in `frontend/package.json` despite the original feature brief referencing "the same SWR patterns" — that brief was wrong about what exists. The plan below adopts a **tiny in-repo SWR-shaped hook** (`useResource`) layered over the existing typed `api` client, so we can refactor `queue/page.tsx` later without churn and without adding `swr` to dependencies. Do not add `swr` to `package.json` unless the user explicitly asks for it.
- **Storage placement is not your concern in this milestone.** Every action this UI exposes goes through an existing backend endpoint under `/v1/admin/curator/*` (see `backend/api/curator.py:75-526`). Do not invent new state or new caches in the frontend — the backend is the system of record (AGENTS.md §3) and the four Redis rules (AGENTS.md §4) are upheld server-side. The frontend's only durable state is the stub `X-User-Email` in `localStorage` (already wired by `frontend/components/UserPicker.tsx`).
- **The never-delete invariant is enforced server-side.** The frontend must never present a "delete skill" or "delete snapshot" affordance. Allowed write affordances in this UI are exactly: `pause`, `resume`, `run`, `run --dry-run`, `rollback`, `restore/{skill_id}`, `pin/{skill_id}`, `unpin/{skill_id}`, `reviews/{id}/approve`, `reviews/{id}/reject`. If you are tempted to add anything else, re-read AGENTS.md §5.
- **Two backend gaps are unavoidable.** Before this UI can fully implement features (3) snapshot browser and (5) review proposals queue grouped by run, the backend needs two thin read-only endpoints. They are listed in Phase 1 below as **prerequisite backend tasks**, scoped narrowly so they do not turn this into an M4 plan. If the implementation agent is forbidden from touching the backend, it must stub the UI sections with an explicit "endpoint not available" placeholder rather than synthesize data.
- **Auth gate must fail closed.** The `/admin/curator/*` route group must early-return a 403-style "admin only" screen for any non-admin principal. Today the frontend has no way to ask "what roles do I have?" — the user picker lets you *claim* to be `admin@org` and the backend trusts the `X-User-Email` header to map that to roles via `roles_for_email` (`backend/core/auth/models.py:59-71`). The cheapest correct gate for the POC is: call `GET /v1/admin/curator/status` on mount; if it returns 403 (the response shape from `require_role("admin")` — see `backend/core/auth/deps.py:64`), render the gate; otherwise the user is admin. Do NOT introduce a new `/v1/me` endpoint in this milestone unless explicitly approved.
- **Dry-run "streams report inline without mutating".** The current `POST /v1/admin/curator/run?dry_run=true` is **not** a streaming endpoint — it returns a `CuratorRunRecord` synchronously after building a full report in memory (`backend/api/curator.py:124-144`). "Streaming" in this UI therefore means: show a busy spinner, on response render the transitions table and a Markdown view of the report. Do not add SSE/WebSocket plumbing. If the user later wants true streaming, that is a separate milestone.

## Feature Description

A new `/admin/curator` section in the Next.js frontend that gives platform admins a single operator surface for the deterministic M2 curator and the M3 LLM-review queue. The UI is read-mostly with a small number of high-friction write affordances (every destructive-feeling action requires a two-step confirmation). It is built on the existing typed `api` client (`frontend/lib/api/client.ts`) extended with a `curator.ts` namespace, Tailwind utility classes (no new design system), and a tiny `useResource` hook for the SWR-shaped fetch/revalidate pattern the existing pages will be migrated to later.

The section is composed of six concerns:

1. **Status dashboard** — `/admin/curator` — current pause state, lock state, last run summary, last snapshot, counts of skills by status bucket (`approved`/`stale`/`archived`/`pinned`), and the last N run records linked to a per-run report viewer.
2. **Run controls** — buttons for `pause`, `resume`, `run`, `run --dry-run`. Pause/resume require a one-click confirm; real `run` requires a two-step confirm with the warning text "this writes a snapshot and may transition skills"; dry-run runs immediately and renders its `CuratorRunRecord` + `REPORT.md` inline.
3. **Snapshot browser** — `/admin/curator/snapshots` — list of snapshots with `name` (the UTC-ISO folder), `captured_at`, `skills_count`, `size_bytes`, and a `Rollback…` button per row. Rollback opens a modal that requires the operator to type the snapshot name to confirm.
4. **Per-skill actions** — a "Skills" tab at `/admin/curator/skills` that lists every skill with status + pinned flag and exposes `Pin`/`Unpin` toggles. Archived skills get a `Restore` button. This is *not* the contributor catalog page; it is the admin-only fleet view.
5. **Review proposals queue (M3)** — `/admin/curator/reviews` — pending proposals grouped by `run_id`, with three viewers: (a) `patch` proposals open a unified-diff or full-replacement viewer, (b) `merge` proposals open a side-by-side viewer showing the N input skills and the proposed umbrella, (c) `keep` proposals are shown inline. Each row has `Approve` / `Reject` actions; reject opens a textarea for the optional reason.
6. **Auth gate** — every page under `/admin/curator/**` is wrapped by `RequireAdmin` (a client component) that probes `GET /v1/admin/curator/status` once on mount and renders either the children or a 403 panel.

## User Story

As a **platform admin** I want a single web page where I can see the curator's current health, run a dry-run, see what it *would* do, and only then promote that to a real run, so I never operate the curator blind.

As a **platform admin** I want to rollback to a specific snapshot through a UI with a two-step confirm, so a mistyped CLI argument cannot wipe out a known-good state.

As a **manager promoted to admin** I want pin/unpin and restore-from-archive controls on a per-skill view, so I can rescue a niche-but-critical skill in two clicks instead of editing a Cosmos doc by hand.

As an **admin reviewing M3 LLM proposals** I want every proposal grouped by run, with a diff/merge viewer, so I can approve a consolidation in seconds and reject obvious mistakes with one click.

As a **non-admin user** who navigates to `/admin/curator` I want a clear "admins only" message rather than a broken page, so I don't file a bug.

## Problem Statement

Today the M2 deterministic curator and M3 LLM-review queue are fully implemented server-side (`backend/api/curator.py:75-526`) but have **zero frontend surface**. The only operator path is `curl` against `/v1/admin/curator/*` or invoking the worker CLI. This makes routine ops (paused? what did the last run do? show me pending review proposals) error-prone and gatekeeps the curator behind a small set of CLI-comfortable operators. It also means M3's review queue — the user-facing payoff of an entire milestone of LLM work — is invisible.

## Solution Statement

Ship a focused admin section under `/admin/curator/**` that:

1. Wraps every page in a `RequireAdmin` client component that fails closed via a `GET /v1/admin/curator/status` probe.
2. Adds a single `curator` namespace to the existing typed `api` client, mapping 1:1 to backend endpoints (no new state machines on the client side).
3. Composes the UI from small Tailwind components (`StatusBadge` is reused; `Confirm`, `Modal`, `MarkdownView`, `DiffView`, `RunRecordCard`, `SnapshotRow`, `ProposalCard` are new under `frontend/components/curator/`).
4. Uses a tiny in-repo `useResource` hook (Promise-cache + revalidate-on-mount + manual `mutate`) so we get SWR ergonomics without adding `swr` to `package.json`. The same hook can be retrofitted onto `queue/page.tsx` and `my-submissions/page.tsx` in a follow-up.
5. Adds two narrow read-only backend endpoints (`GET /v1/admin/curator/snapshots`, `GET /v1/admin/curator/runs`) as **prerequisite Phase 1 tasks** — both are thin wrappers over services that already exist (`snapshot.list_snapshots`, the Blob listing already inlined in `status_endpoint`).
6. Ships frontend tests under `frontend/__tests__/curator/` using **Vitest + Testing Library** (added to `devDependencies`), with a `tests/msw/handlers.ts` mocking every `/v1/admin/curator/*` endpoint. No e2e in this milestone — that is the `e2e-test` skill's job and lands as a separate plan.

### Patterns to Follow

Extracted from `frontend/app/admin/queue/page.tsx` and `frontend/lib/api/client.ts`.

**Naming Conventions:**
- Pages: `page.tsx` per route, default-exported React component named after the page (`ReviewQueuePage`, `CuratorDashboardPage`).
- Components: PascalCase named export from a file of the same name (`StatusBadge.tsx` → `export function StatusBadge`).
- API namespace: `api.curator.status()`, `api.curator.pause()`, etc. — verbs match backend endpoint names.
- Types: TS interfaces in `types.ts`, snake_case JSON field names preserved (e.g. `last_loaded_at`, not `lastLoadedAt`).

**Fetching Pattern (today — replace with `useResource` going forward):**

```tsx
const [rows, setRows] = useState<SkillListItem[]>([]);
const [error, setError] = useState<string | null>(null);
async function load() {
  try { setRows(await api.admin.queue()); setError(null); }
  catch (err) { setError(String(err)); }
}
useEffect(() => { void load(); }, []);
```

**Fetching Pattern (new — `useResource`, used by every new page):**

```tsx
const { data, error, isLoading, mutate } = useResource(
  ["curator", "status"],
  () => api.curator.status(),
);
```

`mutate()` is called after every successful write action (e.g. `await api.curator.pause(); await mutate();`).

**Error Display Pattern (mirror `queue/page.tsx:60-64`):**

```tsx
{error && (
  <div className="rounded border border-rose-300 bg-rose-50 p-3 text-sm text-rose-800">
    {String(error)}
  </div>
)}
```

**Action-Button Busy Pattern (mirror `queue/page.tsx:94-108`):**

```tsx
<button
  disabled={busy}
  onClick={() => void run()}
  className="rounded bg-emerald-600 px-3 py-1 text-white disabled:opacity-50"
>
  Run
</button>
```

**Confirm Pattern (new — used by Run and Rollback):**

`<Confirm title="..." body="..." confirmText="..." onConfirm={fn} />` opens a Tailwind modal. For rollback, require the user to *type* the snapshot name to enable the confirm button (analogous to GitHub repo deletion).

**Other Relevant Patterns:**
- Client Component opt-in: first line of every interactive page is `"use client";`.
- API error shape: server returns `{error_code, message}` (see `types.ts:60-63`). `useResource`'s error path stringifies via `JSON.stringify(body)` already (see `client.ts:25-29`); do not rebuild parsing.
- Local dev identity: there is no auth in local dev beyond the `X-User-Email` header. The `RequireAdmin` gate works locally by switching `UserPicker` to `admin@org`, which `roles_for_email` maps to `["user", "admin"]` because `admin@org` is in `settings.admin_email_set()` per `.env.local`.

## NOTES

**Why a homegrown `useResource` instead of `swr`:** The repo today has zero `swr` usage and is intentionally lean (`frontend/package.json` shows just `next`, `react`, `react-markdown`, `zod`). Pulling in `swr` to support six pages crosses a dependency threshold without buying anything we can't write in ~60 lines. If a future milestone wants `swr`-the-library, swap the implementation behind the same hook signature in one commit.

**Why two backend endpoints in a "frontend" plan:** The snapshot browser and runs viewer cannot exist truthfully without them, and both are 15-20 line wrappers over services that already exist (`snapshot.list_snapshots`, the Blob listing already inlined in `status_endpoint`). Adding them here keeps the milestone honest. If repo policy forbids backend changes in a frontend plan, lift those into a separate Phase 0 plan and mark this one blocked until merge.

**Why MSW v2 + Vitest instead of Jest + nock:** Vitest is the de-facto default for Next 14 in 2024+, integrates with the existing `tsconfig` path aliases with zero custom transform, and MSW v2 mocks at the `fetch` layer — which is exactly where `frontend/lib/api/client.ts` operates. No `next/jest.config.js` indirection required.

**Why no e2e tests in this milestone:** Repo has an `e2e-test` skill (`.opencode/skills/e2e-test/SKILL.md`) that is designed to run after implementation against the live local stack. Hooking it in here would double-scope the plan. Land this, then run the skill in a follow-up.

**Confidence score for one-pass execution: 7/10.** Risk concentrated in: (a) MSW v2 + Vitest config in a project that has never had a test runner — first-time setup is fiddly; (b) the `text/markdown` response for `getRunReport` requires either a `callText` helper or a content-type branch in `call`, both straightforward but easy to miss; (c) the per-skill admin listing depends on a backend endpoint that may or may not exist (`backend/api/admin.py` was not exhaustively re-read here — implementer must verify and add a thin wrapper if absent). All three are flagged inline; an implementer who reads the GOTCHAs should land cleanly.
