## Feature Description

Three new integration tests that close the M2 coverage gap on the load-bearing invariants — rollback safety, janitor message recovery, pinning immunity. Each test runs end-to-end against the local emulator stack (`docker compose up -d`), auto-skips when the stack is down, and asserts both observable state (Cosmos doc fields, Blob contents) and audit trail completeness.

Specifically:

1. **`backend/tests/integration/test_curator_rollback_round_trip.py`** — Proves that after a curator pass mutates state, `rollback(snapshot_name=...)` restores the Cosmos+Blob world to byte-for-byte identity with the pre-pass state, AND that the rollback itself produced a `pre-rollback-{ts}` snapshot so it can be undone.

2. **`backend/tests/integration/test_janitor_sweep.py`** — Proves that the janitor sweep finds Cosmos docs whose `classifier_status='queued'` is older than the staleness threshold, re-pushes their `id` onto the `queue:classifier` Redis LIST exactly once each, and writes one `classify` audit row per re-queue with `metadata.requeued_by='janitor'`.

3. **`backend/tests/integration/test_curator_pin_unpin.py`** — Proves that a pinned skill survives two full curator passes — one at `stale_days+1` age, one at `archive_days+1` age — remaining `status='approved'`; then `POST /unpin/{id}` followed by a third pass at the same archive-eligible age transitions it normally to `archived`.

## User Story

As a **platform admin debugging an over-eager curator run** I want CI to prove that `rollback` restores the published catalog byte-for-byte and is itself reversible, so I can trust the rollback button in production.

As a **platform owner** I want CI to prove that the janitor sweep recovers lost classifier messages exactly once, so the rule-#4 mitigation isn't silently broken by a refactor.

As a **manager pinning a critical skill** I want CI to prove that pinning truly grants immunity against every curator transition rule (not just one), so my pin survives any future planner change.

## Problem Statement

M2 shipped the curator, rollback, janitor, and pinning *code* and the *unit* coverage for the planner and report renderer. The integration test surface, however, only exercises happy-path: a dry-run vs. real-run on a synthetic seeded skill (`test_curator_run.py`), admin endpoint smoke tests (`test_curator_endpoints.py`), and a single usage POST (`test_usage_pipeline.py`). The three M2 acceptance criteria that *cannot* be verified by unit tests — Blob byte-for-byte rollback, end-to-end Redis re-queue, multi-pass pinning immunity across simulated time — have no integration coverage. A regression in any of these would ship green to prod.

Three concrete regressions M2.1 must guard against:

1. **Silent rollback corruption.** A future refactor of `curator_rollback.rollback` (or `snapshot._build_snapshot_tar`) could break byte-equality without anyone noticing — the bytes might still *deserialize* fine but the catalog SHA values stored in Cosmos would diverge from what `/download` returns.
2. **Janitor duplicate or missing re-queue.** A future change to `janitor_classifier_queue` could push messages twice (causing duplicate classification) or skip stale docs (leaving them stuck queued forever). Neither is caught by current tests.
3. **Pin-leak.** A future planner change that adds a new transition rule could forget the `if doc.pinned: skipped.append(...)` guard. The unit truth-table test would still pass for the *existing* rules. Only a multi-rule integration test catches this.

## Solution Statement

Three self-contained integration tests, each in its own file under `backend/tests/integration/`, each carrying `pytestmark = pytest.mark.integration`, each doing its own seed-and-cleanup. No new fixtures, no new test helpers extracted to `conftest.py` — pattern is "duplicate `_cleanup` per test file" matching the existing M2 integration tests (`test_curator_run.py:61-78`, `test_usage_pipeline.py:37-61`, `test_e2e_happy_path.py:61-83`).

### 1. `test_curator_rollback_round_trip.py`

**Setup**
- Seed N=3 approved skill docs into Cosmos via `skills.upsert_item` (mirror `test_curator_run.py:_seed_old_skill` at lines 31-58 but **with `usage.last_loaded_at` set to `now - 200 days`** so the planner will emit `archive` transitions on the real run).
- For each skill, upload a *real bundle blob* to `published/{skill_id}/1.0.0/bundle.tar.gz` (the snapshot service can only snapshot bytes that exist; the planner-only test in `test_curator_run.py` doesn't need real bytes because it asserts at the JSON level — this test does). Use `backend.core.blob.put_published(blob, settings, skill_id=..., version="1.0.0", data=...)` with synthetic content like `b"bundle:" + skill_id.encode()`.
- Capture the pre-pass SHA256 of each `published/` blob into a `dict[str, str]`.
- Capture the pre-pass `status` of each Cosmos doc.
- Capture the pre-pass list of snapshot folders via `snapshot_svc.list_snapshots(blob, settings)`.

**Action**
- Authenticate as admin via `as_admin(app, email="admin@example.com")`.
- `POST /v1/admin/curator/run` (real run). Assert 200. Capture `record["snapshot_name"]` from the response body.
- After the run completes, assert each seeded skill's Cosmos `status` is now `archived` and the blob has been *copied* to `archive/{id}/{ver}/bundle.tar.gz` (M2 leaves the `published/` source in place per defense-in-depth — confirm both prefixes have the bytes).
- `POST /v1/admin/curator/rollback?id={snapshot_name}`. Assert 200. Capture `rollback_result["pre_rollback_snapshot_name"]`.

**Assertions**
- For each seeded skill_id: re-read `published/{id}/1.0.0/bundle.tar.gz` from Azurite and compute SHA256. Assert it equals the captured pre-pass SHA256 for that skill — **byte-for-byte equality**.
- For each seeded skill_id: re-read the Cosmos doc and assert `status == "approved"` (the original status).
- `snapshot_svc.list_snapshots(blob, settings)` now includes the `pre-rollback-{ts}` folder name returned in `rollback_result.pre_rollback_snapshot_name`. (Verifies rollback-is-itself-reversible.)
- Query the `audit` container for each `skill_id`: at least one row with `action="rollback"` exists per skill, and at least one earlier row with `action="archive"` exists per skill.

**Cleanup**
- Delete the three seeded Cosmos `skills` docs + their `audit` rows.
- Delete the three `published/{id}/...` blobs and any `archive/{id}/...` blobs created.
- Delete only the snapshot folders this test created: the run's `snapshot_name`, the rollback's `pre_rollback_snapshot_name`, and any `curator/runs/` report folders whose names match those two run_ids. Use `list_blobs(name_starts_with=f"{folder}/")` and `delete_blob` per blob. **This is the one place test cleanup may call `delete_blob` — production curator code may not.** Add a short comment in the cleanup helper citing `backend/tests/integration/test_usage_pipeline.py:37-56` as the precedent for "tests may delete; production may not."

### 2. `test_janitor_sweep.py`

**Setup**
- Compute the staleness cutoff: `stale_seconds = settings.classifier_blpop_timeout_seconds * settings.janitor_classifier_stale_multiplier`.
- `now = datetime.now(UTC)` (real wall clock — janitor doesn't need a fixed clock since it accepts `now=`).
- Seed N=3 skill docs with: `classifier_status='queued'`, `status='pending'`, `uploaded_at = now - (stale_seconds + 60)` (well past cutoff). Use distinct `skill_id`s.
- Drain the Redis `queue:classifier` LIST at test start: `while await redis.lpop(key_queue_classifier()): pass`.
- Assert the queue is empty.

**Action**
- Call the service function directly: `await janitor_classifier_queue(skills=..., audit=..., redis=..., settings=..., now=now)`. Use `app.state.cosmos_db`, `app.state.redis` (mirror `test_e2e_happy_path.py:101-128`). Capture the return value `result`.

**Assertions**
- `result == {"scanned": 3, "requeued": 3}`.
- `await redis.llen(key_queue_classifier()) == 3`.
- Drain the queue with `lpop` and collect the doc IDs. Assert the set equals `{seeded_doc_ids}` — every seeded doc is on the queue exactly once, no duplicates, no extras.
- Query `audit` for each seeded `skill_id`: exactly one row with `action="classify"` and `metadata.requeued_by == "janitor"` per skill.
- Idempotence sub-assertion: call `janitor_classifier_queue` a second time with the same `now`. Assert `result == {"scanned": 3, "requeued": 3}` (Cosmos still shows `classifier_status='queued'` because the janitor doesn't mutate doc status — confirms re-runs are safe; the classifier worker is the one that flips status on successful processing). The queue now holds 6 messages — 3 from each sweep. This is the documented behavior; the production mitigation against runaway re-queueing is that the classifier worker BLPOPs them faster than the janitor cron interval. Document this in a test comment.

**Cleanup**
- Delete the 3 seeded `skills` docs and their `audit` rows.
- Drain `queue:classifier` again.

### 3. `test_curator_pin_unpin.py`

**Setup**
- Pick `stale_days = settings.curator_stale_days` (default 30) and `archive_days = settings.curator_archive_days` (default 90).
- Choose a fixed `baseline = datetime(2026, 1, 1, tzinfo=UTC)` (well-defined, deterministic).
- Seed N=2 approved skills:
  - `pinned_skill_id`: `pinned=True`, `pinned_by="admin@example.com"`, `usage.last_loaded_at = baseline - 200 days`, `uploaded_at = baseline - 200 days`.
  - `unpinned_skill_id`: `pinned=False`, same timestamps. (Control: this one must transition normally to prove the planner is engaged.)
- Upload real bundle bytes to `published/` for both (so archive copies have something to copy).

**Action A — pass 1, age past stale threshold**
- `now1 = baseline + timedelta(days=stale_days + 1)` (i.e. last_loaded_at is `stale_days + 201` days behind `now1` — past stale, well past archive).
- Call `execute_pass(dry_run=False, ..., now=now1)` directly (service-level test — bypass HTTP because we want to inject `now=`). Capture `record1`.

**Action B — pass 2, age past archive threshold**
- `now2 = baseline + timedelta(days=archive_days + 1)`.
- Call `execute_pass(dry_run=False, ..., now=now2)`. Capture `record2`.

**Assertions after passes A+B**
- Re-read pinned skill from Cosmos: `status == "approved"`, `pinned is True`.
- Re-read unpinned skill from Cosmos: `status == "archived"` (transitioned at pass 1 to stale, pass 2 to archived — or possibly directly to archived at pass 1 depending on planner; either way must be `archived` by end).
- `record1.skipped_pinned` and `record2.skipped_pinned` both contain `pinned_skill_id`.
- No `audit` row exists for `pinned_skill_id` with `action in {"archive", "stale"}`.
- At least one `audit` row exists for `unpinned_skill_id` with `action="archive"`.

**Action C — unpin and re-run at archive-eligible age**
- `POST /v1/admin/curator/unpin/{pinned_skill_id}` as admin. Assert 200 and `response["pinned"] is False`.
- Re-read Cosmos to confirm `pinned == False`.
- `now3 = now2` (same as pass 2 — archive-eligible).
- Call `execute_pass(dry_run=False, ..., now=now3)`. Capture `record3`.

**Assertions after pass C**
- Re-read pinned-then-unpinned skill: `status == "archived"`.
- `record3.skipped_pinned` does NOT contain `pinned_skill_id`.
- One new `audit` row with `action="archive"` exists for `pinned_skill_id`.
- One `audit` row with `action="unpin"` exists for `pinned_skill_id` (written by the `POST /unpin` handler — verify via `backend/api/curator.py:243-258`).

**Cleanup**
- Delete both seeded Cosmos docs + their `audit` rows.
- Delete blobs at `published/{both}/` and `archive/{both}/`.
- Delete the three run report folders created (use the three captured `record.run_id` values).
- Delete the three snapshot folders created (use the three captured `record.snapshot_name` values).

### Patterns to Follow

**Test module docstring (canonical: `test_curator_run.py:1-4`, `test_usage_pipeline.py:1-9`):** Every test file opens with a 2–4 line docstring naming what it asserts and noting "Requires emulator stack. Skipped automatically otherwise."

**Fixture pattern (canonical: `test_curator_run.py:22-28` — verbatim across every M2 integration test):**

```python
@pytest.fixture
async def app_client():
    app = create_app()
    transport = httpx.ASGITransport(app=app)
    async with httpx.AsyncClient(transport=transport, base_url="http://test") as client:
        async with app.router.lifespan_context(app):
            yield client, app
```

Copy this verbatim. Do not extract to `conftest.py` — the M2 tests intentionally duplicate it; duplication signals isolation.

**Seed pattern (canonical: `test_curator_run.py:31-58`):** Open a Cosmos client via `get_cosmos_client(settings)` inside a `try/finally` that closes it; build a `body=` dict matching the `SkillDoc` schema exactly (every required field present, defaults explicit); call `skills.upsert_item(body=...)`. Mirror this exactly — do not invent a fixture helper.

**Cleanup pattern (canonical: `test_curator_run.py:61-78` for Cosmos; `test_usage_pipeline.py:37-61` for Cosmos+Redis):**

```python
async def _cleanup(settings, skill_id: str) -> None:
    cosmos = get_cosmos_client(settings)
    try:
        db = cosmos.get_database_client(settings.cosmos_db_name)
        for name in ("skills", "audit", "usage_events"):
            cont = db.get_container_client(name)
            with contextlib.suppress(Exception):
                async for row in cont.query_items(
                    query="SELECT c.id, c.skill_id FROM c WHERE c.skill_id=@id",
                    parameters=[{"name": "@id", "value": skill_id}],
                    partition_key=skill_id,
                ):
                    with contextlib.suppress(Exception):
                        await cont.delete_item(item=row["id"], partition_key=skill_id)
    finally:
```

## NOTES

**Why three separate files instead of one parameterized test.** Each test has distinct seed/cleanup needs; combining them would obscure failures and slow down focused iteration. The existing M2 integration tests follow the one-file-per-invariant pattern; new tests do the same.

**Why service-direct calls in the pin test and janitor test, but HTTP in the rollback test.** Pin test needs `now=` injection (HTTP `POST /run` doesn't accept a clock override). Janitor test exercises a service function with no HTTP surface (`POST /v1/admin/curator/janitor` exists but the test wants the typed return value `{"scanned": N, "requeued": N}` which is easier to assert against the function return than the JSON response — though either works). Rollback test is HTTP because the admin endpoints ARE part of what M2 ships; verifying them through HTTP exercises FastAPI request validation + auth gating + service wiring in one shot.

**Why corrupting `published/` between curator run and rollback (rollback test).** Without corruption, the byte-for-byte assertion is trivially satisfied — the curator never deletes the source bytes (defense-in-depth). Corrupting `published/` between the curator run and the rollback forces the rollback service to *actively* restore from the snapshot tar; the assertion then meaningfully proves the restoration code works.

**Why idempotence sub-assertion in the janitor test.** Production runs the janitor on a cron schedule; if the same stale doc gets swept twice (because the classifier worker is slow), the queue gets duplicate entries. This is the documented current behavior (the classifier worker is expected to drain the queue between janitor cron ticks). Asserting the second sweep behaves identically locks this behavior in — a future change that, e.g., adds a "don't re-queue if already on queue" guard would change the second result to `{"scanned": 3, "requeued": 0}` and force the test to be updated (or surface a behavioral change that needs a design decision).

**Why no parameterized variant of the pin test for `archived` status pinning.** Out of scope. The current planner only considers `approved` and `stale` as input statuses; pinning an `archived` skill is a no-op for the planner. A separate test could assert this, but it's not in the M2.1 charter.

**Why we don't extract `_cleanup` helpers to `conftest.py`.** The duplication is intentional — every M2 integration test has its own slightly-different seed/cleanup shape (some need Blob cleanup, some don't; some need run_id captures, some don't). Premature extraction would either over-fit one test's needs or grow a bag-of-flags shared helper. The existing tests duplicate; the new ones do too.

**Confidence score for one-pass implementation: 8.5/10.** The production code being exercised is in place and the patterns are well-established by the three existing M2 integration tests. The two main risks: (1) `snapshot.list_snapshots` filtering of `pre-rollback-*` folders — needs verification at impl time, mitigated by direct `blob_client.exists()` fallback; (2) the exact attribute name on `app.state` for the blob service client — needs a 1-line `grep` at impl time. Both are 5-minute verifications, not architectural unknowns.
